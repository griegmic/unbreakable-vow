Approved 16 Implementation Notes: Make A Vow Fast Path

This screen is the returning-user power path. The route can remain quick-vow internally, but the user-facing product language should be Make a vow / Seal a vow. Default behavior: if a saved payment method exists, Stake $50 → opens Apple Pay / Stripe PaymentSheet directly and continues to the sealed moment. Screen 16B is fallback-only for users without a saved method.

Required interactions

  • Vow field is directly editable and visually focused as the primary object.
  • Verdict row opens the approved verdict date sheet.
  • Stake tiles update the amount, cheeky copy, consequence line, and CTA.
  • Other opens custom amount entry and supports $0.
  • Destination line opens the approved cause / destination sheet.

$0 behavior

  • $0 is allowed as an accountability vow.
  • CTA changes to Seal my vow →, not “Stake $0.”
  • Payment step is skipped unless a previous saved card is required for other active staked vows.
  • Copy changes from “If broken, $50 goes…” to a non-money accountability consequence.
  • No Stripe setup should be triggered for a pure $0 vow.

Witness path

  • Returning users default to their most recent witness in a slightly faded row; tapping the row or Change commits/changes the witness and brings opacity to 1.0.
  • The prefilled row uses a gold initial avatar, witness name, contextual subcopy, and a right-side Change affordance.
  • Add/change witness opens native contacts / recent witness picker.
  • If contacts are already synced, recent witnesses appear first.
  • If contacts are not synced, show the approved contact permission sheet only after tapping Add witness.
  • No Decide later path on Quick Vow. Users who want a slower witness decision can use Guided setup.
  • After seal, route to the correct invite state based on whether a named witness exists.

Payment and metadata

  • For paid vows, Apple Pay / Stripe PaymentSheet is the first-class payment path.
  • Saved payment method is reused when available; the primary path skips 16B and fires the payment sheet directly from screen 16.
  • 16B stays in the design system for edge cases: no saved method, lost Stripe customer state, or first paid vow after prior $0-only usage.
  • Selected destination is sent with the vow metadata and reflected in the confirmation/payment screens.
  • CTA copy dynamically mirrors the current amount and setup state.
  • Payment errors produce warning haptic and stay on the same screen with a clear retry path.
Donald check: this is intentionally not mocked as more screens. These are implementation acceptance requirements for the approved fast-path screen.
