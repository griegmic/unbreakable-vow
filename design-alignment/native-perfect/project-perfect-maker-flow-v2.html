Final Design Audit

  • Changed back: Set Stake now matches the stronger last version: large Fraunces amount, $20/$50/$100/Other, contained stake card, mobile-web-like “If you break it” row, compact verdict row.
  • Date: corrected to a compact editable row plus bottom sheet. No standalone date route in the normal flow.
  • Witness: corrected toward the attached design. The native delta is contact selection after tap, with recents and “share link instead.”
  • Auth: OTP and name are split. Mixing them was below the 80% confidence bar because it creates two jobs on one anxious screen.
  • Payment: visual target is Apple Pay-first, but implementation must defer to Stripe PaymentSheet availability. Do not fake Apple Pay if Stripe cannot present it.
  • Celebration: Option A is recommended. It combines “sealed” and “send Joe” into one persistent moment instead of quick disappearing screens.

Implementation Plan Before Build

  • Map every native route to this flow: cinematic, native-guided, native-stake, native-witness, native-auth, native-seal/payment, native-vow-detail, native-dashboard.
  • Keep mobile web copy/layout as baseline. Only implement native-specific deltas: contact picker, haptics, native share, push timing, Apple Pay-first PaymentSheet.
  • Add haptic map: light for chip/stake/date/contact, medium for next/seal/share, success for OTP/card/sealed/kept, warning for invalid/payment failed/broken.
  • Verify all navigation escape hatches: back goes to previous flow step, menu opens, dashboard cards open exact vow, post-seal routes to vow detail/pending witness.
  • Payment QA: Expo Go shows test-safe fallback; TestFlight validates Apple Pay/card setup with Stripe test mode. Confirm no charge at seal, charge only on broken verdict.
  • QA loop: screenshot every native maker state, compare to this mock + live mobile web, fix overlap/readability, then produce a TestFlight checklist for your demo.
